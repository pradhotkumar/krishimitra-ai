# Requirements Document: KrishiMitra AI SaaS Web Application

## Introduction

The KrishiMitra AI SaaS Web Application is a modern, production-ready web platform that provides Indian farmers with visual access to AI-powered agricultural guidance through an intuitive chat interface. The application complements the existing voice-first telephony system by offering a web-based alternative for farmers who have access to smartphones or computers. Built using Next.js 14 with TypeScript and deployed on AWS infrastructure, the platform delivers multilingual support (Hindi and English), real-time AI responses powered by Amazon Bedrock, and a premium user experience with responsive design and agricultural theming.

## Glossary

- **Frontend**: The Next.js 14 web application that renders the user interface and handles client-side interactions
- **Backend**: AWS Lambda functions that process API requests and implement business logic
- **API_Gateway**: AWS API Gateway service that exposes RESTful endpoints for frontend consumption
- **Bedrock**: Amazon Bedrock AI service that generates intelligent responses using Claude 3 Sonnet model
- **DynamoDB**: AWS NoSQL database service that stores messages, sessions, and contact submissions
- **Amplify**: AWS Amplify hosting service that deploys and serves the Next.js application
- **Session**: A conversation context identified by a unique sessionId that maintains message history
- **Message**: A single chat interaction containing user input or AI response
- **Language**: The communication language, either Hindi ('hi') or English ('en')
- **Chat_Handler**: Lambda function that processes chat requests and generates AI responses
- **Contact_Handler**: Lambda function that processes contact form submissions
- **User**: A farmer or visitor interacting with the web application
- **AI_Response**: Text generated by Amazon Bedrock in response to user queries
- **SessionId**: Unique identifier for a conversation session, format: session_[uuid]
- **MessageId**: Unique identifier for a message, format: msg_[uuid]

## Requirements

### Requirement 1: Frontend Page Rendering

**User Story:** As a user, I want to access different pages of the application, so that I can navigate between homepage, chat, about, and contact sections.

#### Acceptance Criteria

1. WHEN a user navigates to the root URL, THE Frontend SHALL render the homepage with hero section, features, and call-to-action
2. WHEN a user navigates to /chat, THE Frontend SHALL render the chat dashboard with message interface
3. WHEN a user navigates to /about, THE Frontend SHALL render the about page with mission and technology information
4. WHEN a user navigates to /contact, THE Frontend SHALL render the contact page with submission form
5. WHEN any page loads, THE Frontend SHALL display the page within 3 seconds on standard broadband connection

### Requirement 2: Chat Message Submission

**User Story:** As a farmer, I want to send messages to the AI assistant, so that I can get answers to my agricultural questions.

#### Acceptance Criteria

1. WHEN a user types a message and clicks send, THE Frontend SHALL submit the message to the API_Gateway
2. WHEN a message is submitted, THE Chat_Handler SHALL validate the message is non-empty and under 5000 characters
3. WHEN validation passes, THE Chat_Handler SHALL invoke Bedrock to generate an AI response
4. WHEN Bedrock returns a response, THE Chat_Handler SHALL store both user message and AI response in DynamoDB
5. WHEN storage succeeds, THE Chat_Handler SHALL return the AI response to the Frontend
6. WHEN the Frontend receives the response, THE Frontend SHALL display the AI message in the chat interface

### Requirement 3: Session Management

**User Story:** As a user, I want my conversation to be remembered across multiple messages, so that the AI can provide contextually relevant responses.

#### Acceptance Criteria

1. WHEN a user sends their first message without a sessionId, THE Chat_Handler SHALL generate a new unique sessionId
2. WHEN a sessionId is generated, THE Chat_Handler SHALL return it in the API response
3. WHEN a user sends subsequent messages with a sessionId, THE Chat_Handler SHALL use the same sessionId
4. WHEN processing a message with a sessionId, THE Chat_Handler SHALL retrieve conversation history from DynamoDB
5. WHEN conversation history is retrieved, THE Chat_Handler SHALL include it in the Bedrock request for context

### Requirement 4: Multilingual Support

**User Story:** As a farmer, I want to communicate in my preferred language (Hindi or English), so that I can understand the information clearly.

#### Acceptance Criteria

1. WHEN a user selects a language in the chat interface, THE Frontend SHALL include the language parameter in API requests
2. WHEN a chat request specifies language 'hi', THE Chat_Handler SHALL use the Hindi system prompt for Bedrock
3. WHEN a chat request specifies language 'en', THE Chat_Handler SHALL use the English system prompt for Bedrock
4. WHEN Bedrock generates a response, THE AI_Response SHALL be in the same language as the user's message
5. WHEN the language parameter is not 'hi' or 'en', THE Chat_Handler SHALL return a 400 error

### Requirement 5: Input Validation

**User Story:** As a system, I want to validate user inputs, so that I can prevent invalid data from being processed.

#### Acceptance Criteria

1. WHEN a chat request has an empty or null message, THE Chat_Handler SHALL return a 400 status code with error message
2. WHEN a chat request has a message exceeding 5000 characters, THE Chat_Handler SHALL return a 400 status code
3. WHEN a chat request has an invalid language value, THE Chat_Handler SHALL return a 400 status code
4. WHEN a chat request has a malformed sessionId, THE Chat_Handler SHALL return a 400 status code
5. WHEN a contact form has an invalid email format, THE Contact_Handler SHALL return a 400 status code
6. WHEN a contact form has a message shorter than 10 characters, THE Contact_Handler SHALL return a 400 status code

### Requirement 6: Data Persistence

**User Story:** As a system, I want to store conversation data, so that I can maintain session history and enable future analytics.

#### Acceptance Criteria

1. WHEN a user message is processed, THE Chat_Handler SHALL store the message in DynamoDB with sessionId, timestamp, role, content, and language
2. WHEN an AI response is generated, THE Chat_Handler SHALL store the response in DynamoDB with the same session context
3. WHEN storing a message, THE Chat_Handler SHALL generate a unique messageId in format msg_[uuid]
4. WHEN storing a message, THE Chat_Handler SHALL set the timestamp to current ISO 8601 formatted time
5. WHEN a message is older than 90 days, THE DynamoDB SHALL automatically delete it using TTL

### Requirement 7: Conversation History Retrieval

**User Story:** As a system, I want to retrieve conversation history efficiently, so that I can provide context to the AI for better responses.

#### Acceptance Criteria

1. WHEN retrieving conversation history, THE Chat_Handler SHALL query DynamoDB using the sessionId as partition key
2. WHEN querying conversation history, THE Chat_Handler SHALL limit results to the most recent 10 messages
3. WHEN conversation history is retrieved, THE Chat_Handler SHALL return messages in chronological order (oldest first)
4. WHEN a session has no previous messages, THE Chat_Handler SHALL return an empty array
5. WHEN retrieving conversation history, THE Chat_Handler SHALL complete the query within 2 seconds

### Requirement 8: AI Response Generation

**User Story:** As a farmer, I want to receive intelligent, contextually relevant responses, so that I can get accurate agricultural guidance.

#### Acceptance Criteria

1. WHEN generating an AI response, THE Chat_Handler SHALL invoke Bedrock with Claude 3 Sonnet model
2. WHEN invoking Bedrock, THE Chat_Handler SHALL include the system prompt matching the user's language
3. WHEN invoking Bedrock, THE Chat_Handler SHALL include conversation history for context
4. WHEN invoking Bedrock, THE Chat_Handler SHALL set temperature to 0.7 and max_tokens to 1024
5. WHEN Bedrock returns a response, THE Chat_Handler SHALL extract the text content from the response
6. WHEN Bedrock invocation fails, THE Chat_Handler SHALL return a 500 error with descriptive message

### Requirement 9: Contact Form Submission

**User Story:** As a user, I want to submit contact inquiries, so that I can reach out to the KrishiMitra team with questions or feedback.

#### Acceptance Criteria

1. WHEN a user submits a contact form, THE Frontend SHALL send name, email, phone, message, and language to API_Gateway
2. WHEN the contact request is received, THE Contact_Handler SHALL validate all required fields are present
3. WHEN validation passes, THE Contact_Handler SHALL generate a unique messageId
4. WHEN a messageId is generated, THE Contact_Handler SHALL store the submission in DynamoDB with status 'pending'
5. WHEN storage succeeds, THE Contact_Handler SHALL return success response with messageId
6. WHEN the Frontend receives success response, THE Frontend SHALL display a confirmation message to the user

### Requirement 10: Error Handling

**User Story:** As a user, I want to receive clear error messages when something goes wrong, so that I understand what happened and can take appropriate action.

#### Acceptance Criteria

1. WHEN any validation error occurs, THE Backend SHALL return a 400 status code with descriptive error message
2. WHEN a Bedrock service failure occurs, THE Backend SHALL return a 500 status code with user-friendly message
3. WHEN a DynamoDB operation fails, THE Backend SHALL log the error and return a 500 status code
4. WHEN an unhandled exception occurs, THE Backend SHALL log the error details and return a generic 500 error
5. WHEN the Frontend receives an error response, THE Frontend SHALL display the error message to the user
6. WHEN any error occurs, THE Backend SHALL log error details to CloudWatch with timestamp and request context

### Requirement 11: API Response Structure

**User Story:** As a frontend developer, I want consistent API response formats, so that I can reliably parse and display data.

#### Acceptance Criteria

1. WHEN a chat request succeeds, THE API_Gateway SHALL return a response with response, sessionId, timestamp, and language fields
2. WHEN any request fails, THE API_Gateway SHALL return a response with error field containing descriptive message
3. WHEN any response is returned, THE API_Gateway SHALL include CORS headers to allow frontend access
4. WHEN any response is returned, THE API_Gateway SHALL set Content-Type header to application/json
5. WHEN a successful chat response is returned, THE sessionId SHALL match the format session_[uuid]

### Requirement 12: Chat UI Functionality

**User Story:** As a user, I want an intuitive chat interface, so that I can easily interact with the AI assistant.

#### Acceptance Criteria

1. WHEN the chat page loads, THE Frontend SHALL display an empty message list and input field
2. WHEN messages exist in the session, THE Frontend SHALL display them in chronological order (oldest to newest)
3. WHEN a user types in the input field, THE Frontend SHALL enable the send button
4. WHEN the input field is empty, THE Frontend SHALL disable the send button
5. WHEN a message is being sent, THE Frontend SHALL display a typing indicator
6. WHEN a response is received, THE Frontend SHALL remove the typing indicator and display the AI message
7. WHEN the user presses Enter in the input field, THE Frontend SHALL submit the message

### Requirement 13: Language Selection

**User Story:** As a user, I want to switch between Hindi and English, so that I can communicate in my preferred language.

#### Acceptance Criteria

1. WHEN the chat interface loads, THE Frontend SHALL display a language selector with Hindi and English options
2. WHEN a user selects Hindi, THE Frontend SHALL update the UI text to Hindi
3. WHEN a user selects English, THE Frontend SHALL update the UI text to English
4. WHEN a user sends a message, THE Frontend SHALL include the selected language in the API request
5. WHEN the language is changed, THE Frontend SHALL update placeholder text and button labels accordingly

### Requirement 14: Responsive Design

**User Story:** As a user, I want the application to work on different devices, so that I can access it from my phone, tablet, or computer.

#### Acceptance Criteria

1. WHEN the application is viewed on a mobile device, THE Frontend SHALL display a single-column layout
2. WHEN the application is viewed on a tablet, THE Frontend SHALL adapt the layout for medium screen sizes
3. WHEN the application is viewed on a desktop, THE Frontend SHALL display the full multi-column layout
4. WHEN the screen size changes, THE Frontend SHALL adjust the layout without requiring a page reload
5. WHEN the chat interface is displayed on mobile, THE Frontend SHALL ensure the input field remains accessible above the keyboard

### Requirement 15: Performance Optimization

**User Story:** As a user, I want fast page loads and quick responses, so that I can efficiently get the information I need.

#### Acceptance Criteria

1. WHEN the homepage is accessed, THE Frontend SHALL achieve First Contentful Paint within 1.5 seconds
2. WHEN any page is accessed, THE Frontend SHALL achieve Time to Interactive within 3 seconds
3. WHEN a chat message is sent, THE API_Gateway SHALL respond within 3 seconds under normal load
4. WHEN a Lambda function is invoked, THE Backend SHALL complete execution within 30 seconds
5. WHEN images are loaded, THE Frontend SHALL use Next.js Image optimization for faster loading

### Requirement 16: Security and Authentication

**User Story:** As a system administrator, I want secure API endpoints, so that unauthorized users cannot abuse the system.

#### Acceptance Criteria

1. WHEN any API request is received, THE API_Gateway SHALL enforce HTTPS protocol
2. WHEN API requests exceed 1000 per second, THE API_Gateway SHALL apply throttling
3. WHEN user input is received, THE Backend SHALL sanitize HTML and script tags before storage
4. WHEN storing sensitive data, THE DynamoDB SHALL encrypt data at rest
5. WHEN CORS requests are made, THE API_Gateway SHALL only allow requests from the configured Amplify domain

### Requirement 17: Unique Identifier Generation

**User Story:** As a system, I want to generate unique identifiers for sessions and messages, so that I can track and retrieve data accurately.

#### Acceptance Criteria

1. WHEN a new session is created, THE Chat_Handler SHALL generate a sessionId in format session_[uuid]
2. WHEN a new message is stored, THE Chat_Handler SHALL generate a messageId in format msg_[uuid]
3. WHEN multiple identifiers are generated, THE Backend SHALL ensure all identifiers are unique
4. WHEN a sessionId is generated, THE Backend SHALL use UUID v4 format
5. WHEN a messageId is generated, THE Backend SHALL use UUID v4 format

### Requirement 18: Logging and Monitoring

**User Story:** As a system administrator, I want comprehensive logging, so that I can monitor system health and troubleshoot issues.

#### Acceptance Criteria

1. WHEN any Lambda function is invoked, THE Backend SHALL log the request ID and timestamp
2. WHEN an error occurs, THE Backend SHALL log error details including name, message, and stack trace
3. WHEN a Bedrock request is made, THE Backend SHALL log the model used, latency, and token usage
4. WHEN a DynamoDB operation is performed, THE Backend SHALL log the operation type and duration
5. WHEN logs are written, THE Backend SHALL use structured JSON format for CloudWatch

### Requirement 19: Homepage Content

**User Story:** As a visitor, I want to see an informative homepage, so that I can understand what KrishiMitra offers and how to use it.

#### Acceptance Criteria

1. WHEN the homepage loads, THE Frontend SHALL display a hero section with product name and tagline
2. WHEN the homepage loads, THE Frontend SHALL display a features section highlighting key capabilities
3. WHEN the homepage loads, THE Frontend SHALL display a call-to-action button to start chatting
4. WHEN the homepage loads, THE Frontend SHALL display information about the voice helpline integration
5. WHEN the homepage loads, THE Frontend SHALL display AWS architecture diagram or description

### Requirement 20: About Page Content

**User Story:** As a visitor, I want to learn about KrishiMitra's mission and technology, so that I can understand the project's goals and approach.

#### Acceptance Criteria

1. WHEN the about page loads, THE Frontend SHALL display the mission statement
2. WHEN the about page loads, THE Frontend SHALL display information about the technology stack
3. WHEN the about page loads, THE Frontend SHALL display information about AWS services used
4. WHEN the about page loads, THE Frontend SHALL display the target audience and use cases
5. WHEN the about page loads, THE Frontend SHALL display contact information or links to other pages

### Requirement 21: Deployment and Hosting

**User Story:** As a developer, I want automated deployment, so that code changes are quickly reflected in the production environment.

#### Acceptance Criteria

1. WHEN code is pushed to the main branch, THE Amplify SHALL trigger an automatic deployment
2. WHEN deployment is triggered, THE Amplify SHALL build the Next.js application
3. WHEN the build succeeds, THE Amplify SHALL deploy the application to production
4. WHEN the deployment completes, THE Amplify SHALL serve the application over HTTPS
5. WHEN the deployment fails, THE Amplify SHALL notify the development team

### Requirement 22: Environment Configuration

**User Story:** As a developer, I want environment-specific configuration, so that I can use different settings for development, staging, and production.

#### Acceptance Criteria

1. WHEN Lambda functions are deployed, THE Backend SHALL read configuration from environment variables
2. WHEN the Frontend is built, THE Frontend SHALL use environment-specific API Gateway URLs
3. WHEN accessing AWS services, THE Backend SHALL use the configured AWS region
4. WHEN accessing DynamoDB, THE Backend SHALL use the configured table names
5. WHEN invoking Bedrock, THE Backend SHALL use the configured model ID

### Requirement 23: API Gateway Configuration

**User Story:** As a system architect, I want properly configured API endpoints, so that the frontend can reliably communicate with the backend.

#### Acceptance Criteria

1. WHEN API Gateway is deployed, THE API_Gateway SHALL expose a POST endpoint at /api/chat
2. WHEN API Gateway is deployed, THE API_Gateway SHALL expose a POST endpoint at /api/contact
3. WHEN API Gateway is deployed, THE API_Gateway SHALL expose a GET endpoint at /api/health
4. WHEN any endpoint is called, THE API_Gateway SHALL route the request to the appropriate Lambda function
5. WHEN any endpoint is called, THE API_Gateway SHALL apply request validation using JSON schema

### Requirement 24: Health Check Endpoint

**User Story:** As a system administrator, I want a health check endpoint, so that I can monitor system availability.

#### Acceptance Criteria

1. WHEN the health endpoint is called, THE API_Gateway SHALL return the current system status
2. WHEN all services are operational, THE API_Gateway SHALL return status 'healthy'
3. WHEN any service is degraded, THE API_Gateway SHALL return status 'degraded'
4. WHEN critical services are unavailable, THE API_Gateway SHALL return status 'unhealthy'
5. WHEN the health check runs, THE API_Gateway SHALL include status of Bedrock, DynamoDB, and Lambda services

### Requirement 25: Message Display Formatting

**User Story:** As a user, I want messages to be clearly formatted, so that I can easily distinguish between my messages and AI responses.

#### Acceptance Criteria

1. WHEN displaying a user message, THE Frontend SHALL apply distinct styling to differentiate it from AI messages
2. WHEN displaying an AI message, THE Frontend SHALL apply distinct styling to differentiate it from user messages
3. WHEN displaying any message, THE Frontend SHALL show the timestamp
4. WHEN displaying messages, THE Frontend SHALL ensure proper text wrapping for long content
5. WHEN displaying Hindi text, THE Frontend SHALL use appropriate fonts that support Devanagari script

### Requirement 26: Error Recovery

**User Story:** As a user, I want the system to handle errors gracefully, so that temporary failures don't prevent me from using the application.

#### Acceptance Criteria

1. WHEN a Bedrock throttling error occurs, THE Backend SHALL retry the request with exponential backoff
2. WHEN a DynamoDB throttling error occurs, THE Backend SHALL retry the operation up to 3 times
3. WHEN a network timeout occurs, THE Backend SHALL return a 503 status code with retry-after header
4. WHEN the Frontend receives a 503 error, THE Frontend SHALL display a message asking the user to try again
5. WHEN retries are exhausted, THE Backend SHALL return a final error response

### Requirement 27: Rate Limiting

**User Story:** As a system administrator, I want rate limiting, so that the system is protected from abuse and excessive usage.

#### Acceptance Criteria

1. WHEN requests from a single IP exceed 100 per minute, THE API_Gateway SHALL return a 429 status code
2. WHEN the burst limit of 200 requests is exceeded, THE API_Gateway SHALL throttle additional requests
3. WHEN rate limiting is applied, THE API_Gateway SHALL include a Retry-After header in the response
4. WHEN the Frontend receives a 429 error, THE Frontend SHALL display a message about rate limiting
5. WHEN rate limiting is triggered, THE API_Gateway SHALL log the event to CloudWatch

### Requirement 28: Data Model Validation

**User Story:** As a developer, I want strict data model validation, so that only valid data is stored in the database.

#### Acceptance Criteria

1. WHEN storing a message, THE Backend SHALL validate that sessionId matches the pattern session_[uuid]
2. WHEN storing a message, THE Backend SHALL validate that role is either 'user' or 'assistant'
3. WHEN storing a message, THE Backend SHALL validate that language is either 'hi' or 'en'
4. WHEN storing a message, THE Backend SHALL validate that content is non-empty and under 5000 characters
5. WHEN storing a contact submission, THE Backend SHALL validate that email matches standard email format

### Requirement 29: Bedrock Model Configuration

**User Story:** As a system architect, I want consistent AI model configuration, so that responses maintain quality and performance standards.

#### Acceptance Criteria

1. WHEN invoking Bedrock, THE Chat_Handler SHALL use the Claude 3 Sonnet model
2. WHEN invoking Bedrock, THE Chat_Handler SHALL set temperature to 0.7
3. WHEN invoking Bedrock, THE Chat_Handler SHALL set top_p to 0.9
4. WHEN invoking Bedrock, THE Chat_Handler SHALL set max_tokens to 1024
5. WHEN invoking Bedrock, THE Chat_Handler SHALL use anthropic_version 'bedrock-2023-05-31'

### Requirement 30: System Prompt Management

**User Story:** As a product manager, I want well-crafted system prompts, so that the AI provides relevant agricultural guidance.

#### Acceptance Criteria

1. WHEN using Hindi language, THE Chat_Handler SHALL use a system prompt that instructs the AI to respond in Hindi
2. WHEN using English language, THE Chat_Handler SHALL use a system prompt that instructs the AI to respond in English
3. WHEN any language is used, THE Chat_Handler SHALL include instructions about agricultural topics (crops, weather, schemes, prices)
4. WHEN any language is used, THE Chat_Handler SHALL instruct the AI to provide practical, actionable advice
5. WHEN any language is used, THE Chat_Handler SHALL instruct the AI to keep responses concise (2-3 paragraphs)

### Requirement 31: Frontend State Management

**User Story:** As a user, I want the chat interface to maintain state correctly, so that my conversation persists during my session.

#### Acceptance Criteria

1. WHEN a user sends a message, THE Frontend SHALL add the message to the local state immediately
2. WHEN a response is received, THE Frontend SHALL add the AI message to the local state
3. WHEN the page is refreshed, THE Frontend SHALL clear the local message state
4. WHEN a sessionId exists, THE Frontend SHALL store it in localStorage for persistence
5. WHEN the user returns to the chat page, THE Frontend SHALL retrieve the sessionId from localStorage

### Requirement 32: Accessibility Compliance

**User Story:** As a user with disabilities, I want the application to be accessible, so that I can use it with assistive technologies.

#### Acceptance Criteria

1. WHEN any page is rendered, THE Frontend SHALL use semantic HTML elements
2. WHEN interactive elements are present, THE Frontend SHALL ensure they are keyboard navigable
3. WHEN images are displayed, THE Frontend SHALL provide alt text descriptions
4. WHEN form inputs are present, THE Frontend SHALL provide associated labels
5. WHEN color is used to convey information, THE Frontend SHALL ensure sufficient contrast ratio (4.5:1 minimum)

### Requirement 33: Loading States

**User Story:** As a user, I want visual feedback during loading, so that I know the system is processing my request.

#### Acceptance Criteria

1. WHEN a message is being sent, THE Frontend SHALL display a loading indicator
2. WHEN waiting for an AI response, THE Frontend SHALL display a typing indicator
3. WHEN a page is loading, THE Frontend SHALL display a loading spinner or skeleton screen
4. WHEN an API request is in progress, THE Frontend SHALL disable the send button to prevent duplicate submissions
5. WHEN a response is received, THE Frontend SHALL remove all loading indicators

### Requirement 34: Navigation

**User Story:** As a user, I want easy navigation between pages, so that I can access different sections of the application.

#### Acceptance Criteria

1. WHEN any page is displayed, THE Frontend SHALL show a navigation header with links to all main pages
2. WHEN a navigation link is clicked, THE Frontend SHALL navigate to the corresponding page
3. WHEN on the homepage, THE Frontend SHALL highlight the home link in the navigation
4. WHEN on any page, THE Frontend SHALL display a footer with additional links and information
5. WHEN on mobile devices, THE Frontend SHALL provide a hamburger menu for navigation

### Requirement 35: Contact Form Validation

**User Story:** As a user, I want real-time form validation, so that I can correct errors before submitting.

#### Acceptance Criteria

1. WHEN a user enters an invalid email, THE Frontend SHALL display an error message below the email field
2. WHEN a user enters a message shorter than 10 characters, THE Frontend SHALL display an error message
3. WHEN required fields are empty, THE Frontend SHALL disable the submit button
4. WHEN all fields are valid, THE Frontend SHALL enable the submit button
5. WHEN the form is submitted successfully, THE Frontend SHALL clear all fields and show a success message

### Requirement 36: Token Usage Tracking

**User Story:** As a system administrator, I want to track AI token usage, so that I can monitor costs and optimize performance.

#### Acceptance Criteria

1. WHEN a Bedrock response is received, THE Chat_Handler SHALL extract input token count from the response
2. WHEN a Bedrock response is received, THE Chat_Handler SHALL extract output token count from the response
3. WHEN token counts are extracted, THE Chat_Handler SHALL log them to CloudWatch
4. WHEN storing a message, THE Chat_Handler SHALL optionally include token usage in the metadata field
5. WHEN token usage is logged, THE Chat_Handler SHALL include the model name and timestamp

### Requirement 37: Session Expiry

**User Story:** As a system administrator, I want automatic data cleanup, so that old conversation data doesn't accumulate indefinitely.

#### Acceptance Criteria

1. WHEN a message is stored in DynamoDB, THE Backend SHALL set an expiresAt attribute to 90 days in the future
2. WHEN the expiresAt time is reached, THE DynamoDB SHALL automatically delete the message
3. WHEN a contact submission is stored, THE Backend SHALL not set an expiry (contacts are retained)
4. WHEN TTL deletion occurs, THE DynamoDB SHALL not charge for the delete operation
5. WHEN messages are expired, THE DynamoDB SHALL remove them within 48 hours of the expiry time

### Requirement 38: CORS Configuration

**User Story:** As a frontend developer, I want proper CORS configuration, so that the frontend can make API requests without being blocked.

#### Acceptance Criteria

1. WHEN any API response is returned, THE API_Gateway SHALL include Access-Control-Allow-Origin header
2. WHEN a preflight OPTIONS request is received, THE API_Gateway SHALL return allowed methods and headers
3. WHEN CORS headers are set, THE API_Gateway SHALL allow Content-Type and Authorization headers
4. WHEN CORS headers are set, THE API_Gateway SHALL allow GET, POST, and OPTIONS methods
5. WHEN in production, THE API_Gateway SHALL restrict Access-Control-Allow-Origin to the Amplify domain

### Requirement 39: Conversation Context Limits

**User Story:** As a system architect, I want to limit conversation context size, so that API requests remain efficient and cost-effective.

#### Acceptance Criteria

1. WHEN retrieving conversation history, THE Chat_Handler SHALL limit results to 10 most recent messages
2. WHEN building the Bedrock request, THE Chat_Handler SHALL include only the retrieved history
3. WHEN conversation history exceeds 10 messages, THE Chat_Handler SHALL exclude older messages from context
4. WHEN a session has fewer than 10 messages, THE Chat_Handler SHALL include all available messages
5. WHEN conversation history is empty, THE Chat_Handler SHALL send only the current message to Bedrock

### Requirement 40: Infrastructure as Code

**User Story:** As a DevOps engineer, I want infrastructure defined as code, so that I can version control and reproduce the deployment.

#### Acceptance Criteria

1. WHEN deploying the backend, THE Infrastructure SHALL be defined using Terraform or AWS SAM templates
2. WHEN deploying Lambda functions, THE Infrastructure SHALL specify memory, timeout, and environment variables
3. WHEN deploying DynamoDB tables, THE Infrastructure SHALL specify partition keys, sort keys, and indexes
4. WHEN deploying API Gateway, THE Infrastructure SHALL specify endpoints, methods, and integrations
5. WHEN infrastructure changes are made, THE Infrastructure SHALL be versioned in Git
